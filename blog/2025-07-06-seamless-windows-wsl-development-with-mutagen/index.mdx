# Seamless Windows-WSL development with Mutagen

The holy grail of Windows development: native filesystem performance on both Windows and Linux, with
real-time synchronization between them.

This setup enables you to leverage Windows tools where they excel while maintaining blazing-fast
filesystem I/O in WSL for Python, machine learning, and other Linux-native workloads. GPU
virtualization ensures you can run CUDA and DirectX workloads efficiently.

The approach almost completely circumvents the cross-platform filesystem performance bottleneck.

<!-- truncate -->

## Introduction

Windows is used by [28%](https://2025.stateofdevs.com/en-US/technology/#desktop_os) of all
developers. And since not every tool runs on Windows, they may also need a unix-based operating
system on the side.

Usually, any [popular](https://distrowatch.com/dwres.php?resource=popularity) linux distro will do.
Personally I prefer something [Debian](https://www.debian.org/) based like
[Ubuntu](https://ubuntu.com/desktop). Or [Pop!OS](https://system76.com/pop/) which is based on
Ubuntu.

The most popular way to run Linux on Windows is using the
[Windows Subsystem for Linux](https://docs.microsoft.com/en-us/windows/wsl/about). WSL2 provides a
full Linux kernel and allows running Linux applications natively on Windows. It has become the de
facto standard for Windows developers who need Linux tools.

## The problem with WSL filesystem performance

While WSL2 has revolutionized Windows development, it comes with an inhibiting limitation:
**cross-filesystem I/O is painfully slow**.

Reading Windows files from WSL (via `/mnt/c/`) results in terrible performance. Reading WSL files
from Windows (via `\\wsl.localhost\`) is slightly better, but also pretty bad.

For developers working with large codebases, Python environments, or machine learning datasets, this
performance gap is a deal-breaker. And even for smaller projects, the difference is very noticeable.

### Benchmarking the performance gap

To quantify this issue, I created a
[benchmark](https://github.com/webbertakken/wsl-filesystem-benchmark) for comparing native
filesystem performance against cross-filesystem operations.

<Image
  img={require('./assets/filesystem-performance-comparison.webp')}
  alt="Bar chart comparing filesystem performance between Windows and WSL"

The results are stark: while both Windows and WSL achieve excellent performance on their native
filesystems (100% baseline), cross-filesystem operations tell a different story:

- WSL accessing Windows filesystem: ~9% of native performance
- Windows accessing WSL filesystem: ~12% of native performance

These numbers represent averaged results across various file sizes and operations. In practical
terms, this means operations like `npm install`, `git status`, or Python package installations can
take 10-20x longer when crossing filesystem boundaries.

## The solution

We can completely circumvent the performance bottleneck by following a simple rule

- 💡 When working in Windows, use the Windows filesystem
- 💡 When working in Linux, use the Linux filesystem

This leaves us with a new problem:

- ❌ Changed files are only reflected on one side
- ❌ Diverging filesystems

Both of which we can solve

- ✅ Real-time cross-system file synchronisation

## Alternatives

Before getting into the details, you might ask; why not just use VMs or dual boot? It's a fair
question.

In fact, I have evaluated a few categories of alternatives, before settling on a WSL-based solution
with synchronisation.

### Type 2 hypervisors

- ❌ **VMware Workstation Pro**: The pro version allows using host APIs to access the GPU and
  provides 3D acceleration, but not true passthrough (PCIe-level access). This limitation can be
  significant for machine learning workloads that expect direct GPU access.
- ❌ **QEMU/KVM VFIO**: QEMU on Windows 11 typically runs without hardware-assisted virtualization
  (like KVM), limiting its efficiency. Windows 11 lacks direct support for VFIO, inhibiting true GPU
  passthrough.

Since there is no passthrough, we would not be able to work with LLMs and other GPU workloads
effectively.

### Type 1 Hypervisors

- ⚠️ **VMware ESXi**: The free version has limited GPU passthrough support. The paid version
  supports GPU passthrough (Direct Device Assignment), although no official support for consumer
  GPUs.
- ⚠️ **Hyper-V**: Microsoft's Hyper-V does offer GPU passthrough (Discrete Device Assignment),
  although no official support for either Windows 11 or consumer GPUs.
- ✅ **Proxmox**: Offers built-in PCI passthrough and works well for this task

Looks like Proxmox is an excellent candidate bare metal hypervisor that allows you to run Windows
and Linux alongside each other and assign hardware to either of them

Caveat: To switch, Shut down VM #1 → detach GPU → attach GPU to VM #2 → start VM #2.

:::info

For the gamers among us, this approach would also incur a small performance loss (1-5% loss in FPS).

:::

### Dual Boot

- ✅ **Native performance**: Files live natively where they're accessed most.
- ❌ **Context switching**: Rebooting between OSes disrupts flow.
- ❌ **Shared state issues**: Managing shared files and configurations between OSes is messy.

A dual boot setup is probably a good option if you intend to do all development work on Linux and
other tasks on Windows that are not needed at the same time.

Personally I like to alternate between playing games and developing stuff on the regular.

### WSL2 + Sync (This approach)

- ✅ **Native performance**: Files live natively where they're accessed
- ✅ **Unified development environment**: No context switching, instant tool access
- ⚠️ **GPU Virtualization**: WSL2 provides GPU acceleration using a virtual GPU driver. It runs CUDA
  and DirectX workloads (AI/ML workloads, graphics rendering, etc.) efficiently.

For sake of moving forward I made the assumption that GPU virtualisation will be sufficient for my
development needs.

What remains then is to solve the cross-filesystem performance issue with real-time synchronisation.

## Enter Mutagen: real-time bidirectional sync

The solution is surprisingly elegant: keep files native on both filesystems and synchronise them in
real-time. This is where Mutagen excels.

[Mutagen](https://mutagen.io/) is a real-time file synchronization tool designed for development
workflows. Unlike traditional sync tools, it's built specifically for handling development files
with features like:

- **Real-time synchronization** with cross-platform filesystem watching
- **Conflict resolution** for simultaneous edits, creation, deletes
- **Extensive ignore patterns** for node_modules, build artifacts, etc.

Mutagen places a filesystem watcher on both the Windows and WSL sides. When you save a file on
Windows, the watcher detects the change and syncs it to WSL within the same second. The same happens
in reverse—changes in WSL appear instantly on Windows. This includes new files, modifications, and
deletions.

Because each environment accesses files on its native filesystem, you get full native performance.
The synchronisation happens so quickly that it feels like you're working with a single, unified
filesystem.

I evaluated several synchronisation solutions before choosing Mutagen:

- **rsync**: Doesn't support bidirectional sync with proper conflict resolution
- **Unison**: Requires identical versions on both endpoints and complex configuration
- **rclone bisync**: Designed for cloud storage, not optimised for rapid local changes
- **Syncthing**: Looked promising but doesn't support two folders on the same host
- **Native Windows tools**: Not designed for development workflows

Mutagen stands out with its development-focused features:

- Intelligent conflict resolution
- Comprehensive ignore patterns
- Minimal CPU and memory overhead
- Symbolic link support
- Permission handling

## Setting up the perfect sync

Let's walk through setting up a robust, production-ready sync between Windows and WSL2.

### Prerequisites

- Windows 10/11 with WSL2 enabled
- Ubuntu 22.04 (or your preferred WSL distro) installed
- Administrative access to Windows

### Step 1: install Mutagen

Download Mutagen from [mutagen.io/downloads](https://mutagen.io/downloads) and install it on
Windows. Add it to your PATH for easy command-line access.

Create a `.mutagen.yml` file in your Windows home directory (`C:\Users\YourName\.mutagen.yml`):

### Step 3: create the sync session

Now we'll create the actual sync between your Windows and WSL repositories:

```powershell
# Start the Mutagen daemon
mutagen daemon start

# Create the sync session
mutagen sync create `
  "C:\Users\$env:USERNAME\Repositories" `
  "\\wsl.localhost\Ubuntu-22.04\home\$env:USERNAME\Repositories" `
  --name="repositories"

# Verify sync creation
mutagen sync list
```

<Image
  img={require('./assets/sync-creation-success.png')}
  alt="PowerShell terminal showing successful sync creation with green status indicators"
/>

The initial sync may take several minutes depending on how many repositories you already have in
your `~/Repositories` folder.

You can monitor progress:

```powershell
# Watch sync progress in real-time
mutagen sync monitor repositories
```

In my case I had 130 repositories with loads of files, so it was taking a while.

Turns out you can skip discovering files and force a synchronisation cycle using the `flush` option.

```powershell
mutagen sync flush repositories
```

Now the initial sync was very quick.

### Step 4: Automate startup

To ensure Mutagen starts automatically when Windows boots, create a startup script:

<details>
  <summary>Create startup script</summary>

```powershell
# register-startup.ps1
$startupFolder = "$env:APPDATA\Microsoft\Windows\Start Menu\Programs\Startup"
$mutagenPath = (Get-Command mutagen).Path
$startupScript = @"
@echo off
mutagen daemon start
mutagen sync resume repositories
"@

# Create startup batch file
$scriptPath = "C:\Users\$env:USERNAME\Bin\mutagen-startup.bat"
New-Item -ItemType Directory -Path (Split-Path $scriptPath) -Force | Out-Null
Set-Content -Path $scriptPath -Value $startupScript

# Create shortcut in startup folder
$WshShell = New-Object -comObject WScript.Shell
$Shortcut = $WshShell.CreateShortcut("$startupFolder\Mutagen.lnk")
$Shortcut.TargetPath = $scriptPath
$Shortcut.Save()

Write-Host "Mutagen will now start automatically on login" -ForegroundColor Green
```

</details>

With Mutagen in place, the development experience is transformed:

- **Native performance**: Each environment accesses files at full native speed

## Conclusion

While the ideal solution would be native cross-filesystem performance in WSL, the current reality
requires workarounds. Mutagen provides an elegant solution that maintains the benefits of both
Windows and WSL without the complexity of hypervisors or the disruption of dual booting.

For developers working with mixed technology stacks—particularly those combining traditional Windows
development with modern Python/ML workflows—this approach offers the best balance of performance,
usability, and simplicity.

Whether you're working with Python, Node.js, machine learning, or just want the flexibility to use
the best tools from both ecosystems, this setup provides the foundation for a truly modern
development environment.
